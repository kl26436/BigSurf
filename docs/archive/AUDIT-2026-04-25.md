# Big Surf Workout Tracker — code & UX audit

**Date:** 2026-04-25
**Scope:** Code quality / architecture, performance & bundle health, bugs & correctness, UX / design-system drift
**Method:** Four parallel deep-read passes over the repo, cross-checked against [CLAUDE.md](CLAUDE.md) rules
**Status of findings:** All citations are real file:line references; a few are flagged as "verify before fixing" because the surrounding callsites weren't fully traced.

---

## Executive summary

The codebase is in good overall shape — strong token discipline, clean BEM naming, consistent module layout, and a documented design system that's mostly being followed. The drift that exists is concentrated in three places:

1. **Performance hot paths in active-workout flow.** Two changes — parallelizing exercise autofill on workout start, and not rebuilding the entire wizard `innerHTML` on every keystroke — would account for most of the user-perceived sluggishness during a session.
2. **A couple of real correctness risks** in PR detection, plate calculator unit handling, and rapid-save data-loss potential. None are catastrophic but all are user-facing.
3. **Mechanical copy and token cleanups** — high count, near-zero risk to fix.

The two highest-leverage changes are:
- Parallelize the autofill loop in `active-workout-ui.js` (~1–2s faster workout start)
- Add `limit(500)` to `loadAllWorkouts` in `data-manager.js` (caps dashboard cold-load cost)

## Section 1 — correctness & data risks

These are bugs or data-integrity risks. Ordered by severity.

### 1.1 — Data loss on overlapping `setDoc` writes  *(severity: high)*

**File:** `js/core/data/data-manager.js:158`

```javascript
await withRetry(() => withTimeout(setDoc(docRef, savedDoc)));
```

`setDoc` without `{ merge: true }` overwrites the entire document. If two writes overlap (rapid set logging, debounce edge cases, retry firing alongside a fresh save), the later write wins and earlier in-flight data is dropped.

**Failure mode:** User logs three sets in quick succession; set 2's data is lost when set 3's write fires before set 2's completes.

**Fix:**
- Pass `{ merge: true }` to all `setDoc` calls that update an in-flight workout doc, OR
- Confirm every save path goes through `debouncedSaveWorkoutData` (a single owner of the write).

**Verify before fixing:** Trace whether the current save path is actually a full-doc replace or a partial update. If `savedDoc` is reconstructed from full state every save, the practical risk is lower than the code shape suggests.

### 1.2 — PR detection ignores `originalUnit`  *(severity: high)*

**File:** `js/core/features/pr-tracker.js:428–439`

`processWorkoutForPRs` calls `checkForNewPR(exerciseName, set.reps, set.weight, equipment)` and writes `unit: set.originalUnit || 'lbs'` separately. The `set.weight` value is compared directly against `currentPRs.maxWeight.weight`, but those two values may have different `originalUnit`s.

**Failure mode:**
- User logs 100 lb max → next session, logs 50 kg (≈110 lb) → PR is missed because `50 < 100`.
- Inverse: user logs 50 kg max → logs 100 lb (≈45 kg) → false PR fires because `100 > 50`.

**Fix:** Normalize both sides to a canonical unit (lbs or kg) before the comparison:

```javascript
const toLbs = (w, unit) => unit === 'kg' ? w * 2.20462 : w;
const setLbs = toLbs(set.weight, set.originalUnit || 'lbs');
const prLbs = toLbs(currentPRs.maxWeight.weight, currentPRs.maxWeight.unit || 'lbs');
```

### 1.3 — Plate calculator may double-convert weight  *(severity: medium-high — safety)*

**File:** `js/core/features/plate-calculator.js:300–341` (`openPlateCalcPopover`)

The popover reads `unit` from `AppState.exerciseUnits[exerciseIndex] || AppState.globalUnit`, then converts the input to kg if `isKg`. But the saved set may have a different `originalUnit` than the current popover unit.

**Failure mode:** Set was logged in lbs; popover opens in kg; the lb value is treated as kg and converted again. User loads roughly half the intended weight per side. This has *physical safety implications* — wrong plate breakdowns at the bar.

**Fix:** Read `set.originalUnit` and convert relative to that, not the popover's display unit.

### 1.4 — `resetLocationState()` not called on workout cancel  *(severity: medium)*

**File:** `js/core/workout/workout-session.js:905–962` (`cancelWorkout`)

`cancelWorkout` resets `AppState` and clears timers, but does not import or call `resetLocationState()` from `js/core/features/location-service.js`. Module-scope location state (currentLocation, locationLocked, etc.) leaks across workouts.

**Failure mode:** Start workout A at Gym X → cancel → start workout B → workout B is auto-tagged with Gym X without GPS detection running again.

**Fix:**
```javascript
import { resetLocationState } from '../features/location-service.js';
// ...inside cancelWorkout, before AppState.reset():
resetLocationState();
```

The same fix probably belongs in `completeWorkout` if it isn't already called there — verify.

### 1.5 — UTC bug in body-weight freshness check  *(severity: medium)*

**File:** `js/core/utils/weight-calculations.js:75`

```javascript
const ageMs = Date.now() - new Date(entry.date + 'T00:00:00').getTime();
```

`new Date('2025-01-15T00:00:00')` parses as **local** midnight; `Date.now()` is UTC milliseconds. The math works (both Date methods normalize to UTC under the hood) but the returned `ageMs` is offset by your timezone offset relative to a same-day comparison. Near midnight or right after DST transitions, "is the entry from today?" can resolve incorrectly.

**Failure mode:** Bodyweight freshness prompt fires (or fails to fire) on the wrong day boundary.

**Fix:** Use `AppState.getTodayDateString()` and string-compare YYYY-MM-DD dates, the way the rest of the app handles dates.

### 1.6 — 3-hour timeout / auto-complete logic is missing  *(severity: medium)*

**File:** `js/core/app-initialization.js:753–868` (`showInProgressWorkoutPrompt`)

[CLAUDE.md](CLAUDE.md) under "In-Progress Workout Detection" specifies:
- 3-hour timeout: resume banner only shows for workouts <3h old
- Auto-complete: workouts >3h with exercises done are auto-completed
- Auto-delete: workouts >3h with no exercises are deleted

None of this logic is implemented in the prompt path. The resume banner shows indefinitely.

**Failure mode:** User abandons a workout, forgets, comes back days later → resume banner offers a stale workout with stale rest-timer assumptions, stale GPS state, and a date field that may no longer match.

**Fix:** In `checkForInProgressWorkoutEnhanced()`, before showing the banner:

```javascript
const ageH = (Date.now() - new Date(workoutData.startedAt).getTime()) / 3600000;
if (ageH > 3) {
  if (Object.keys(workoutData.exercises || {}).length > 0) {
    await autoCompleteWorkout(workoutData);
  } else {
    await deleteWorkout(workoutData.id);
  }
  return;
}
```

Also clamp `Math.max(0, ageH)` to defend against device clock skew (`startedAt` in the future).

### 1.7 — Schema migration: untested, fragile, weak ID entropy  *(severity: medium)*

**File:** `js/core/data/schema-migration.js`

Three issues stack up here:

1. **No tests.** The migration is irreversible and operates on user data; zero coverage for old-schema detection, ID-collision handling, partial-failure recovery.
2. **Weak random suffix at line 15.** `Math.random().toString(36).substring(2, 8)` — ~31 bits of entropy. The rest of the codebase already uses `crypto.getRandomValues` (`data-manager.js:57–61`); reuse that helper.
3. **No idempotency.** `runMigration` (lines 94–130) doesn't check whether the target document already exists before writing/deleting. If the tab closes mid-batch, a second run can duplicate. The `needsMigration` check at lines 30–50 looks at ID format only — a doc with the new ID but missing `version` will be treated as already-migrated.
4. **Silent error swallowing at lines 46–49.** `catch (error) { console.error(...); return false; }` makes "migration not needed" indistinguishable from "migration check failed due to network error" upstream.

**Fix path:**
- Add `tests/unit/schema-migration.test.js` covering: legacy detection, ID-collision, partial-failure idempotency, version-field absence.
- Replace `Math.random()` with the `crypto.getRandomValues` helper.
- In `needsMigration`, also flag docs missing `version` or with `version < '3.0'`.
- In `runMigration`, check target existence before delete; consider wrapping each migration in a Firestore transaction.
- Return `{ ok: false, reason }` instead of bare `false` so callers can distinguish.

### 1.8 — Rest-timer `visibilitychange` listener never torn down  *(severity: low)*

**File:** `js/core/workout/rest-timer.js:25–35`

The document-level `visibilitychange` listener that re-syncs active timers persists across workout sessions. If timer state from a previous session is still in module scope when a new workout's timer DOM is mounted, a tab visibility change can re-sync stale state into fresh DOM.

**Fix:** Tag timers with a session ID and ignore the resync when the IDs don't match, or remove the listener on `cancelWorkout` / `completeWorkout`.

`getCurrentPosition()` can resolve with `(0, 0)` in some browser/error states. Currently treated as a valid location.

**Fix:**

if (coords.latitude === 0 && coords.longitude === 0) {
  return { location: null, isNew: false, coords: null };

### 1.10 — Loose validation lets garbage into Firestore  *(severity: low)*

**File:** `js/core/utils/validation.js:38–45`

```javascript
reps: Math.max(0, Math.min(999, Number(set.reps) || 0)),
weight: Math.max(0, Math.min(9999, Number(set.weight) || 0)),
```

Clamps but doesn't reject. A typo of `1000000` becomes 9999 and silently saves; a 0-rep set is silently saved as a non-set. Affects 1RM calc, volume stats, and PR detection downstream.

**Fix:** Reject in strict-validate mode, with a soft toast that asks the user to re-enter, instead of silently clamping.

## Section 2 — performance hot paths

Items most likely to account for "the app feels sluggish sometimes" are flagged.

### 2.1 — Sequential autofill blocks workout start  *(impact: high — user-felt)*

**File:** `js/core/workout/active-workout-ui.js:2050–2055`

```javascript
for (let i = 0; i < AppState.savedData.exercises.length; i++) {
  await loadAutofillForExercise(i);
}
```

Each call hits Firestore (`limit(30)` query in `getLastSessionDefaults`). 8 exercises = 8 sequential round-trips. On a typical mobile network this is the 1–2 second hang between tapping "Start workout" and seeing populated cards.

**Fix:** Single line.

```javascript
await Promise.all(
  AppState.savedData.exercises.map((_, i) => loadAutofillForExercise(i))
);
```

This is the single highest-leverage perf change in the audit.

---

### 2.2 — `renderAll` rewrites entire wizard `innerHTML` on every update  *(impact: high — user-felt)*

**File:** `js/core/workout/active-workout-ui.js:132–146`

Every set update path (`awUpdateSet`, `awAddSet`, set-completion toggle, notes change) calls `renderAll()`, which does `container.innerHTML = \`...\`` with full template interpolation. Inside the post-render `requestAnimationFrame`, `container.querySelectorAll('.aw-notes__textarea').forEach(...)` runs.

**Failure mode:** Keystroke lag in weight/reps inputs, particularly on mid-tier Android devices. The keyboard appears to "lag behind" your taps.

**Fix path:** Bigger refactor than 2.1 — split rendering into stable header + per-exercise body + sticky footer; patch only the changed set row on update. Keep `renderAll` for the initial mount and structural changes (add/remove exercise, reorder).

This is the main keyboard-feel issue.

---

### 2.3 — `loadAllWorkouts` is unbounded  *(impact: high — grows over time)*

**File:** `js/core/data/data-manager.js:776–777`

```javascript
const q = query(workoutsRef, orderBy('date', 'desc'));
```

No `limit()`. Pulls every workout you've ever logged on every dashboard cold-load. Cost grows linearly in your workout count; the user feels this as the app getting slower over time.

**Fix:** Add `limit(500)` to the query (more than you'd ever scroll on dashboard) and switch the deep history view to a paginated load. One-line change for the immediate fix.

---

### 2.4 — Reorder click listeners stack  *(impact: medium)*

**File:** `js/core/workout/exercise-ui.js:121–154`

`addReorderButtons()` calls `addEventListener('click', …)` inside a `cards.forEach` loop, with no removal step. Toggling reorder N times means N listeners per card.

**Failure mode:** Mid-workout reorder mode toggling makes the card respond to clicks slowly, because every previous handler runs.

**Fix:** Switch to event delegation on the container, or run a `removeReorderButtons()` before each `addReorderButtons()` that removes by stored handler ref.

---

### 2.5 — Other unbounded queries  *(impact: medium)*

**File:** `js/core/data/data-manager.js`
- Line 890: `recoverCorruptedWeights` — `getDocs(workoutsRef)` with no limit.
- Lines 942, 952: `countReassignmentImpact` — two scans of full collections.
- Lines 1025–1027: `reassignEquipment` loads all workouts into memory before batching writes.

These are rare paths so they don't drive day-to-day slowness, but at >5,000 workouts they'll become user-felt during the equipment-swap flow.

**Fix:** Add `limit(1000)` per page and loop with progress.

---

### 2.6 — `_lastSessionCache` has no eviction  *(impact: medium)*

**File:** `js/core/data/data-manager.js:184–195`

The session-level cache for "last session defaults" only clears on workout complete. If a user explores a long template with many exercise swaps, the cache grows unbounded mid-session.

**Fix:** LRU cap at ~50 entries, or 5-minute TTL.

---

### 2.7 — Non-batched `style.setProperty` reflow on rest timer tick  *(impact: low)*

**File:** `js/core/workout/active-workout-ui.js:286–294`

The 1Hz `setInterval` calls `fillEl.style.setProperty(...)` each tick, which forces a reflow. Acceptable on modern phones; noticeable on older devices over a 30-minute workout (battery and slight jank).

**Fix path:** Replace JS ticking with a CSS `animation` running over the rest-timer duration; let `transform: scaleX(...)` do the work on the GPU.

---

### 2.8 — CSS `@import` chain blocks first paint  *(not flagged in first-pass — verify)*

**File:** `styles/index.css`

The codebase imports modular CSS files via native `@import`. Browsers fetch `@import`s sequentially per cascade level, so first paint waits on the chain. If your blank-screen / loading-screen feels long on cold launch, this is a candidate.

**Verify first** with the Network and Performance tabs in DevTools on a throttled connection. If the waterfall shows the CSS chain as the long pole, two options:
1. Replace `@import` with a series of `<link rel="stylesheet">` in `index.html` — browser parallelizes those.
2. Concat at deploy time. Contradicts the "no bundler" design rule but is the right answer if the chain measurably hurts.

---

### 2.9 — Two monoliths  *(impact: low — maintainability, not perf)*

- `js/core/workout/exercise-ui.js` — ~2,700 LOC; renders, drag-reorders, swipes-to-delete, plays videos, loads history, draws progress charts.
- `js/core/workout/active-workout-ui.js` — ~2,000 LOC; the wizard plus rest timer, supersets, equipment swap, exercise add.

Not a perf issue today, but they're where most of the perf items in this section live, and a split would make the targeted fixes (2.2, 2.4) much safer.

## Section 3 — architecture & code quality

### 3.1 — Data layer imports UI  *(severity: medium)*

**File:** `js/core/data/data-manager.js:19`

```javascript
import { showNotification, convertWeight, escapeHtml } from '../ui/ui-helpers.js';
```

The data layer should not know about toasts or HTML escaping. The current arrangement creates bidirectional coupling between `data/` and `ui/`.

**Fix:**
- Move `convertWeight` and `escapeHtml` into a dedicated `utils/` module (you already have `weight-calculations.js` — could absorb conversion).
- Have data-manager return result objects (`{ ok, error, data }`); let the caller (UI layer) decide whether to toast.

The same critique applies to `js/core/features/manual-workout.js:11` — feature modules shouldn't import UI helpers either. Same fix shape.

---

### 3.2 — Three weight-conversion implementations  *(severity: medium)*

- `js/core/ui/ui-helpers.js:67` — `convertWeight`, `displayWeight`
- `js/core/utils/weight-calculations.js` — set-total helpers
- `js/core/features/training-insights.js:14` — inline `normalizedWeight` with its own lbs↔kg constants

A precision tweak in one will drift from the others.

**Fix:** Pick one home (suggest `js/core/utils/weight-calculations.js`); export `convertWeight`, `displayWeight`, `normalizedWeight` from it; have ui-helpers and training-insights import from there.

---

### 3.3 — `JSON.parse(JSON.stringify(...))` deep-clone scattered  *(severity: medium)*

Six callsites:
- `js/core/workout/active-workout-ui.js:1773`
- `js/core/workout/workout-session.js:156, 300`
- `js/core/data/data-manager.js:116`
- (plus a couple more in `workout-management-ui.js` based on grep)

The pattern silently coerces `Date` and Firestore `Timestamp` instances to strings. Anywhere downstream that does `x instanceof Date` on a cloned object fails silently.

**Fix:** Use the modern `structuredClone` (supported everywhere you target), or write a `cloneWorkout(w)` helper that preserves Dates.

---

### 3.4 — `main.js` `window.x = x` wall  *(severity: medium)*

**File:** `js/main.js` (~850 lines, mostly window assignments)

Every interactive function the inline `onclick` handlers in `index.html` need is hoisted onto `window`. There's no audit path to detect dead bindings (rename a function in JS, the corresponding `onclick` becomes a silent no-op). The export list is also where unused exports go to die.

**Fix path:**
```javascript
window.api = { startWorkout, completeWorkout, /* ... */ };
window.callApi = (name, ...args) => {
  if (!(name in window.api)) {
    console.error(`API ${name} not bound`);
    return;
  }
  return window.api[name](...args);
};
```

Then update `index.html` to call `onclick="callApi('startWorkout', 1)"`. Renames will fail loudly the first time you click instead of silently.

---

### 3.5 — Bare `console.log` in production code  *(severity: low)*

**File:** `js/core/utils/capacitor-push.js:24, 33, 54, 67, 73, 77, 101`

The codebase rule is `debugLog` (gated on `?debug` URL param). These bare `console.log` calls leak to every user.

**Fix:** Replace with `debugLog`.

### 3.6 — Inline-style scattered  *(severity: low)*

CLAUDE.md design rule 8 prohibits static inline styles in JS template strings. Spot-checks were mostly clean — `composition-detail-ui.js` lines 78, 179 use the *correct* pattern (CSS custom properties for genuinely dynamic values), and the loading-screen toggles in `app-initialization.js` are acceptable. No high-volume violations detected.

**Action:** None needed in current code; keep an eye on new code.

## Appendix — verification commands

These are the lint passes from CLAUDE.md, useful as a pre-merge habit.

```bash
# Title-case violations (filter out classes, ids, comments, tests)
grep -nE '[A-Z][a-z]+ [A-Z][a-z]+' index.html js/ | grep -v 'class=\|id=\|//\|test\|spec'

# "Please" in user-facing strings
grep -rn "Please " js/ index.html

# Three-dot ellipsis instead of …
grep -rn '\.\.\.' js/ index.html | grep -v '//\|spread'

# "Are you sure" anti-pattern
grep -rn 'Are you sure' js/

# "successfully" filler
grep -rn 'successfully' js/

# "cannot" instead of "can't"
grep -rn 'cannot' js/ | grep -v '//'

# Raw color literals in pages/components CSS (excluding tokens.css)
grep -rn 'rgb\|#[0-9a-fA-F]\{3,6\}' styles/pages/ styles/components/ | grep -v tokens.css

# Raw font-size px/rem outside tokens
grep -rnE 'font-size:\s*[0-9]' styles/ | grep -v tokens.css

# Bare console.log (should be debugLog)
grep -rn 'console\.log' js/ | grep -v debugLog

# Firestore writes outside withRetry
grep -rn 'setDoc\|updateDoc\|deleteDoc' js/core/data/ js/core/features/
```
