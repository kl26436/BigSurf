# Equipment Domain — Deep Dive (2026-07-03)

> **⚠️ Historical snapshot — the PRE-8b state.** The target architecture in §6 was executed 2026-07-03/04 (see [ux-overhaul-plan-2026-07.md](ux-overhaul-plan-2026-07.md) Phase 8b): `equipmentId` is dual-written + backfilled, the PR store is id-keyed, `equipment.locationIds[]` exists, and the dual location model + all healing jobs described in §1–§5 were **deleted**. Keep this doc as the record of *why* 8b happened — do not treat §1–§5 as current architecture.

Full map of the equipment system: data model, write paths, UI surfaces, workflows, and a recommended target architecture. Companion to [ux-overhaul-plan-2026-07.md](ux-overhaul-plan-2026-07.md) (which holds the P1 editor-flow bug batch). File:line refs verified against current code; the specs in `specs/` are aspirational and partially unexecuted — trust this doc over them.

## The one-sentence diagnosis

**Equipment has no stable identity**: the mutable `name` string is the de facto foreign key across workouts, templates, and PRs; the equipment↔gym relationship is stored in two disagreeing places reconciled by fire-and-forget healing jobs; and six different "add equipment" flows and ~8 row renderings grew around that unstable core. Every bug reported so far (edit flow breaks, misleading trends, duplicate machines) is a symptom of this.

---

## 1. Data model (as it actually is)

```
users/{uid}/equipment/{id}         ← canonical docs; almost nothing references the ID
  name        "Brand Line — Function" (regenerated by migration, NOT on manual edit)
  brand / line / function          (the function→machine rename in specs never happened)
  equipmentType                    (11 values incl. deprecated 'Machine' bucket)
  baseWeight / baseWeightUnit / plateSides
  locations: string[]              ← gym NAMES — dual-model half #1
  exerciseTypes: string[]          ← exercise NAMES this machine serves
  exerciseVideos / exerciseSettings (per-exercise, keyed by exercise name)
  aliases: string[]                ← 3rd naming layer; written by orphan-linking only,
                                      read by orphan-scanning only
  catalogRef                       ← only on catalog-promoted docs
  notes / createdAt / lastUsed / version (2 or 3)

users/{uid}/locations/{id}.equipment[]  ← {catalogRef, nickname, notes} — dual-model half #2

equipmentCatalog/{brandSlug}       ← admin-seeded Firestore + static 731-machine JS fallback

workouts/templates: exercises[].equipment = NAME STRING  ← the FK problem
PR store: exercisePRs[exerciseName][equipmentName]        ← same, two levels deep
AppState._cachedEquipment          ← invalidated manually at ~15 call sites, no event system
```

**Key consequences:**
- Editing brand/line/function via the identity sheet (`selectFieldValue`/`saveEquipmentField`) does NOT regenerate `name` and does NOT cascade — history and PRs silently orphan.
- `reassignEquipment` (data-manager.js:950-1219) is the only cascade-aware write in the domain (workouts + templates + PRs) — the model for what every identity edit should do, or better, what IDs make unnecessary.
- Three naming layers (`name`, `function`, `aliases[]`) have inconsistent read coverage: `getOrCreateEquipment` dedupes on `name` only; the picker displays `function || name` but pre-selects by `name`; orphan-scan unions all three. A name that resolves on one surface dead-ends on another.

## 2. Write paths (18 found)

Six creation flows, all writing the same doc through different forms:

| # | Flow | Where | Catalog-aware? | Dedup? |
|---|---|---|---|---|
| 1 | Add-equipment wizard (brand→line→function) | library header | yes | `getOrCreateEquipment` fuzzy |
| 2 | Quick-add sheet (catalog multi-select) | gym detail, aw footer | yes | catalogRef guard |
| 3 | Custom form inside quick-add | nested in #2 | no | `getOrCreateEquipment` |
| 4 | `awQuickAddEquipment` bare form | active workout | **no** | **NONE — calls `saveEquipment` directly, bypassing dedup. Prime duplication vector.** |
| 5 | `addEquipmentFromPicker` (name/location/video text inputs) | exercise-library add flow | no | `getOrCreateEquipment` |
| 6 | `commitCatalogAdd` one-tap | catalog browse/search | yes | catalogRef only — no ownership check (the reported add-vs-edit bug) |

Mutations are mostly bespoke `updateDoc` calls per field (identity sheet, base weight, notes, locations, exercise assignment) — no shared "update equipment" helper, no cascade. Healing jobs (`healDuplicateLocationEquipment`, `migrateLocationCatalogRefs`) re-run every session gated only by in-memory Sets; the V3 migration is one-shot per user with no re-run affordance, so post-migration entropy (path #4, identity edits) accumulates forever.

## 3. UI surfaces (10) and redundancy counts

Surfaces: library landing (My gyms / Library / Catalog tabs) · gym detail · equipment detail (restructured: hero + 3 sheets) · catalog machine detail (**orphaned dead code** — its one caller redirects before it renders) · quick-edit sheet · two active-workout equipment sheets (`awOpenEquipmentSheet` + `openSharedEquipmentSheet`, near-duplicate row code) · legacy equipment-picker modal (structurally isolated, oldest visual system) · location-detail equipment list · scan-review (history reconciliation, not a photo scan — no camera flow exists).

Redundancy: **~8 distinct "equipment row" renderings, 3 "pick a gym" UIs, 6 add flows.** The equipment-picker modal (equipment-picker.js + workout-management-ui.js:494) has zero navigation edges to the rest of the equipment universe.

## 4. Workflow friction (walkthroughs)

- **Mid-workout add**: 4-5 taps happy path, but THREE different add surfaces reachable from the same moment with no routing cue.
- **Fix a typo**: quick-edit sheet exists (2 taps mid-workout) but its Name field only edits the composed display string — a brand/line typo forces "Full details" anyway, defeating the sheet's own stated purpose (equipment-quick-edit.js:3-7). And "Full details" currently loses the race (see plan P1 batch).
- **Set up a new gym**: no "+ Add a gym" affordance exists (spec'd in equipment-library-redesign-brief.md:110-115, never built) — gyms only come from GPS-stamping a workout or as a side effect of quick-add. Chicken-and-egg for pre-planning a gym.
- **Fix a video mapping**: ~5 taps, works — but "+ Assign" renders legacy pre-restructure markup (`.equip-detail-page`), the one un-migrated sub-flow off the new detail page.
- **Browse gym X**: 2 taps, clean. The gym-detail redesign works.

Dead ends: location-detail → equipment detail is one-way (back lands at library root); catalog brand-drill tap = fire-and-forget toast while catalog search tap = different function; `renderCatalogMachineDetail` unreachable.

## 5. Ranked problems

1. 🔴 **Name-string as foreign key** — identity edits orphan history/PRs silently (§1).
2. 🔴 **Dual location model** — two write systems, one-way bridge (`promoteCatalogToEquipment`), healing jobs papering drift; legacy-only docs invisible to catalog "already tagged" checks → duplicates.
3. 🔴 **Six add flows / no dedup on the most-used one** — `awSaveNewEquipment` bypasses `getOrCreateEquipment` entirely.
4. 🟡 **Quick-edit sheet misses its 80% case** (brand/line/function not editable there).
5. 🟡 **Isolated equipment-picker modal** — fourth selection UI, free-text gym field, oldest markup.
6. 🟡 **No "+ Add a gym"**; catalog tap inconsistency; orphaned catalog detail page; legacy assign-exercise markup.
7. 🟢 **Vocabulary debt**: `function`→`machine` rename never executed; deprecated `Machine` type still offered; `aliases[]` half-integrated; manual cache invalidation at 15+ sites.

## 6. Target architecture

**Principle: the equipment doc ID is the only identity; everything else is a label.**

- `exercises[].equipmentId` in workouts/templates (keep `equipmentNameAtTime` for display-only history).
- PR store keyed `exercisePRs[exerciseName][equipmentId]` (highest-blast-radius change — pr-tracker.js has 6+ string-index sites).
- One location relationship: `equipment.locationIds[]` (+ optional `location.equipmentIds[]` read cache) — by ID, never name. Delete both legacy halves AND all five bridge/healing functions, which only exist because the halves can disagree.
- One identity-edit function that regenerates `name` from brand/line/machine — and with ID-keying, needs no cascade at all.
- One add primitive: the catalog quick-add sheet with contextual params (gymName, exerciseName, onDone) — retire the bare aw form and the picker's inline add.
- One selection primitive: `openSharedEquipmentSheet` replaces the equipment-picker modal.
- Bundle the `function`→`machine` rename into the ID migration (same write sites; careful: the catalog schema legitimately uses `machine` for a different object).
- Retire `aliases[]` (orphan-linking becomes "write equipmentId onto those history entries").

**Migration order (safe, incremental):**
1. Additive dual-write: every equipment-selection path also writes `equipmentId`. Zero risk.
2. Backfill job (`equipmentMigrationV4`): resolve name strings → IDs using the existing V3 fuzzy/catalog/alias matching; unresolved entries feed the existing scan-review UI (already the right UX for this).
3. Flip consumers ID-first with name fallback, one file at a time (pr-tracker, equipment-planner, plate-calculator, aggregators' machine picker).
4. Collapse the location dual-model (independent; can parallel #3).
5. Delete healing jobs + aliases after telemetry shows zero fresh drift for a release cycle.
6. Reclassify the deprecated `Machine` type bucket last (cosmetic).

## 7. Suggested sequencing vs the UX plan

The P1 editor-flow bugs (race, catalog add-vs-edit, return context — in the plan) fix Kevin's immediate pain and should ship first; they're compatible with this target. The UI consolidation items (§5 #3-6: one add primitive, retire the picker modal, quick-edit gets brand/line rows, + Add a gym, assign-exercise sheet conversion) form a **Phase 8a (M)** that's pure consistency-contract work. The ID migration (§6) is **Phase 8b (L)** — the deepest structural investment in the plan, but it retires the healing jobs, the duplication vectors, AND unblocks the equipment-aware trends (Phase 1) from ever being corrupted by renames. Do 8a before 8b only if user-visible polish is more urgent; do 8b first if data integrity is (recommended: 8b steps 1-2 are additive and can start immediately alongside anything).
